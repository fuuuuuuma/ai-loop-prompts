ループを安全に回すには「ゴール・1ターンの粒度・ガードレール・停止条件」をプロジェクト側に書いておくのが近道。そのまま貼って育てられる雛形です。

CLAUDE.md Claude Code 用。1リポジトリ＝1ループの運用ルールを置く想定。 
# <プロジェクト名> — ループ運用ルール

## ゴール（このループの終了条件）
- 「達成」とみなす状態を1行で書く
- 例: 全テストが green / 比較表の空欄がゼロ / 新着がN回連続でゼロ

## 1ターンでやること
- 状態を確認 → 差分を1つ処理 → 結果を1行で記録、を繰り返す
- 1ターンで触るのは「1ファイル / 1項目」まで（暴走防止）
- 各ターン冒頭に進捗を、末尾に「やったこと / 次にやること」を残す

## ガードレール（やってはいけないこと）
- 取り消せない操作（削除・上書き・force push・公開・課金・外部送信）は必ず確認
- ゴールと無関係な改善・リファクタを足さない
- 同じ修正を2回以上繰り返したら、止めて人間に報告する

## 記録の残し方
- 進捗は progress ログへ追記（次ターンが読めるように）
- 重要な判断・方針確定は memory/ に1ファイル1事実で残す

## 停止したら
- 「達成 / タイムアウト / 復旧不能エラー」のどれかを明示
- 達成したこと・残タスクをサマリで出す

# AGENTS.md — エージェント編成

## このリポジトリの目的
- <何を、どこまで自動で回すかを1行で>

## 共通ルール
- 日本語で応答 / パスは絶対パス / 取り消せない操作は確認
- 出力は他者レビュー前提の品質で仕上げる
- スコープ外のファイルは触らない

## エージェント分担
- explorer : 調査・現状把握（読み取り専用）
- worker   : 実装・修正（1ターン1差分）
- reviewer : 検証・品質チェック（破壊的変更の前段ゲート）

## ループの回し方
1. explorer が残タスクを洗い出す
2. worker が1つだけ処理する
3. reviewer が結果を検証（OK→次へ / NG→worker に差し戻し）
4. 残タスクがゼロ、または改善が頭打ちで停止

## 停止条件
- 全タスク完了 / 2巡連続で新規ゼロ / 復旧不能エラー
- 停止時は結果サマリと残課題を必ず出す

## 禁止
- 認証情報のコミット（.env / credentials.* は .gitignore 必須）
- ゴールと無関係な変更
