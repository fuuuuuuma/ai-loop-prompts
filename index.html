# <プロジェクト名>

## 概要（What / Why）
- 何のプロジェクトか・技術スタック・誰のためかを2〜3行で
- このリポジトリで Claude に任せたい役割（例: 実装・テスト・調査）

## コマンド（Commands）
- セットアップ: `npm install`
- 開発:        `npm run dev`
- テスト:      `npm test`（単体: `npm test -- <path>`）
- Lint / 型:   `npm run lint` / `npm run typecheck`
- ビルド:      `npm run build`

## プロジェクト構造（Structure）
- `src/`   … アプリ本体
- `tests/` … テスト
- `docs/`  … 仕様・設計メモ
- モノレポは各パッケージに個別 CLAUDE.md（編集ファイルに近い方が優先）

## コーディング規約（Conventions）
- フォーマッタ・命名・インポート順などプロジェクト固有のルール
- コメントは「なぜ」を書く。周囲のスタイルに合わせる

## ワークフロー（How to work）
- 着手前にプランを出す → 小さく変更 → テストで確認 → コミット
- 変更は指示スコープ内のみ。ついで改善はしない
- 完了時に「変えた箇所 / 触っていない箇所 / 要確認」を添える

## やってはいけないこと（Don't）
- 取り消せない操作（削除・force push・公開・課金・外部送信）は要確認
- 認証情報をコミットしない（.env / credentials.* は .gitignore）
- 推測で数値・固有名詞を埋めない。不明は「未確認」と書く

## ループ運用（任意・/loop を回すとき）
- ゴール（終了条件）を1行で明記。例「全テスト green」「空欄ゼロ」
- 1ターン＝状態確認→1件処理→記録。同じ失敗を2回繰り返したら停止
- 定期実行は「前回以降の差分だけ」を扱い暴走・重複を防ぐ

## 参照（References）
- 詳細規約は `@docs/conventions.md` を @import で分割読み込み
- パス限定の追加ルールは `.claude/rules/` に置く

# md-doctor — 設定ファイル健康診断

## 使うとき
- CLAUDE.md / AGENTS.md が長くなった・古くなった・新メンバーが迷うとき

## 手順
1. 対象ファイルを読む（無ければ本ページの雛形から作成）
2. 次の観点で診断する
   - 長さ  : 200行超なら分割を提案（@import / .claude/rules/）
   - 普遍性: タスク固有の指示が混じっていないか
   - コマンド: setup / test / build が実在し最新か（package.json と突合）
   - 構造  : 概要→コマンド→構造→規約→Do/Don't→参照 の順か
   - 鮮度  : 廃止ツール・切れたパス・旧体制の記述が無いか
   - 安全  : 認証情報・禁止操作の注意が書かれているか
3. 指摘を「重大度 / 該当行 / 修正案」で一覧化する
4. 承認を得てから反映する（破壊的変更は必ず確認）
5. 末尾に「変えた箇所 / 触っていない箇所 / 要確認」を出す

## やらないこと
- 内容を勝手に増やさない（短く保つのが目的）
- プロジェクト固有の事実をでっち上げない（不明は TODO で残す）
